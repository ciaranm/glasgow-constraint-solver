# Restarts, nogoods, and dom/wdeg weighting

This document describes the search-side machinery added under issue #315:
restarts on a growing cutoff, a `Nogoods` constraint fed reduced nld-nogoods
learned across restarts, and dom/wdeg variable-ordering with dynamic
constraint weighting. The three are separable but interlock — restarts make
the weighting and nogoods worthwhile — so they are documented together.

Everything here is **opt-in**. With no restart schedule and the default
brancher, `solve()` is byte-for-byte the single exhaustive pass it always was;
the restart cutoff is "infinite", no weighting observer is attached, and no
`Nogoods` constraint is installed. Read `state-and-variables.md` and
`constraints.md` first; this doc assumes the propagation/proof framework.

Code map (cited throughout):

- `gcs/restarts.{hh,cc}` — `RestartSchedule` (the cutoff sequence).
- `gcs/solve.cc` — the restart driver loop and `solve_with_state` recursion,
  including reduced-nld extraction.
- `gcs/solve.hh` — `SolveCallbacks::restarts`.
- `gcs/variable_weighting.{hh,cc}` — `WeightingState`, `VariableWeighting` and
  the concrete schemes.
- `gcs/innards/conflict_observer.hh` — the `ConflictObserver` seam.
- `gcs/constraints/nogoods.{hh,cc}` — the `Nogoods` constraint and `NogoodStore`.
- `gcs/search_heuristics.{hh,cc}` — `variable_order::dom_wdeg`, `branch_with`,
  the value generators.

## The restart loop

A single recursive depth-first search, `solve_with_state` in `solve.cc`, serves
all three modes (decision, enumeration, optimisation) with and without restarts
— there is no duplicated search loop. The mechanism is a three-valued return:

```cpp
enum class SearchResult { Complete, Stop, RestartCutoffHit };
```

`Complete` means the subtree was fully explored (carry on with siblings/parent);
`Stop` means a callback asked search to end (unwind to the caller); the new
`RestartCutoffHit` is a third unwind signal — "abandon this subtree, back to the
root, but we are *not* done". It replaces the old `bool` (`Complete`=true,
`Stop`=false) and mirrors the Glasgow Subgraph Solver's searcher.

A `RestartState` (conflicts seen this pass, the cutoff, an enabled flag) is
threaded through the recursion. The cutoff is checked at node entry; the conflict
counter is incremented on every dead end (a propagator wipeout *or* an objective
bound failure). When the budget is spent the node returns `RestartCutoffHit`,
which propagates up frame by frame to the root.

The driver in `solve_with` is then just:

```cpp
do {
    restart.conflicts_since_restart = 0;
    search_result = solve_with_state(0, ..., restart, nogood_store.get(), {}, ...);
    if (search_result == SearchResult::RestartCutoffHit) {
        ++stats.restarts;
        if (auto observer = propagators.conflict_observer())
            observer->on_restart();          // weighting decays/smooths
        restart_schedule->advance();         // grow the cutoff
        restart.cutoff = restart_schedule->current_cutoff();
    }
} while (search_result == SearchResult::RestartCutoffHit);
```

Completeness holds because the cutoff grows without bound **and** the recorded
nogoods stop already-refuted regions being re-explored: a later pass searches
differently (weights and the incumbent objective bound persist), and the growing
Luby cutoff eventually exceeds the whole tree so a final pass completes.

### RestartSchedule

`gcs/restarts.hh`. A value type wrapping a cutoff sequence; v1 is Luby only,
generated by Knuth's reluctant-doubling recurrence, scaled by a unit (default
`luby(100)`):

```cpp
auto sched = RestartSchedule::luby(/* scale = */ 100);
sched.current_cutoff();   // conflicts allowed this pass
sched.advance();          // step to the next term
```

The restart policy is **a tunable, not a design input** — nogood recording,
weight persistence, and the proof lifecycle are all policy-agnostic. If any of
them depended on the policy the design would be wrong (see the `feedback`
memory on tunables). Pick the default, auto-tune later.

`SolveCallbacks` carries `std::optional<RestartSchedule> restarts` (default
`nullopt`); set it to turn restarts on.

## dom/wdeg weighting

The variable-ordering heuristic minimises `dom(x) / W(x)`, where `W(x)` is a
*dynamic* weighted degree that grows as constraints cause conflicts. The write
side and the read side reach the live weighting by two different paths.

### The ConflictObserver write seam

`gcs/innards/conflict_observer.hh`. An abstract `ConflictObserver` is borrowed by
`Propagators` for the duration of a search (`set_conflict_observer` /
`conflict_observer`). Whenever a propagator wipes out a domain,
`Propagators::propagate` calls

```cpp
note_conflict(int constraint_index,
              const vector<SimpleIntegerVariableID> & scope,
              const optional<ReasonFunction> & reason,
              const State & state);
```

exactly once, with the failing constraint's dense index and scope. `on_restart()`
rides the same seam (it is how `solve_with` reaches the live weighting, which it
does not own, at a restart boundary). The observer is deliberately write-only —
no query API — and lives on `Propagators`, not `State`, because a conflict is a
propagation event and this keeps the seam in one place.

### Weighting schemes

`gcs/variable_weighting.hh`. `VariableWeighting : ConflictObserver` adds
`weighted_degree_of`, `on_restart`, and `snapshot`/`load`. Concrete schemes:

- `ClassicDomWDeg` — Boussemart et al. (ECAI 2004): `+1` per wipeout, `W(x)` sums
  `w(c)` over `x`'s constraints with `|fut(c)| >= 2`.
- `RefinedWeighting` — Wattez et al. (ICTAI 2019): per-`(constraint, variable)`
  local weights, incremented by a `Variant` factor (`ia / ca / id / cd / ca.cd`).
  `ca.cd` (`1 / (|fut| * (1 + |dom|))`) is **the default**; its degree-at-root
  equals classic, so selector tie-break tests are unaffected.
- `ConflictHistorySearch` — Habet & Terrioux: recency-weighted ERWA scores with
  `on_restart` smoothing (inert until restarts actually fire).

`DenseConstraintWeighting` factors the dense storage and the read/snapshot/load
machinery shared by the global schemes; `RefinedWeighting` is local so it stores
its own maps. The top-level `WeightingScheme` enum dispatches in
`variable_order::dom_wdeg`.

### WeightingState: the parallel-ready edge format

`WeightingState` is a portable, copyable, `ConstraintID`-keyed value object with
`snapshot` / `load` / `merge` (Sum/Max/Average). It is **only** a seed-in /
snapshot-out edge format — it never enters the engine; only the live
`VariableWeighting`'s observer pointer does. This is what makes share-nothing
parallel sync (per-thread `State`+`Propagators`, periodic lock-free weight
exchange) and proof-mined weight seeding possible without it being built yet.

### The brancher

`variable_order::dom_wdeg(problem | vars, scheme = ca.cd, initial = nullopt)`
returns a `BranchVariableHeuristic` — a `function` called **once per search**
that creates the live weighting, seeds it via `load` if an initial state was
given, attaches it with `propagators.set_conflict_observer`, and returns a
selector that captures it. (Branchers used to be rebuilt per node, which would
have wiped stateful weights; `solve_with` now builds once and reuses the
selector.) The comparison is `dom(a)·W(b) < dom(b)·W(a)` in `double`, with
`-degree_of` as the tie-break. Default brancher is still `dom_then_deg`;
`dom_wdeg` is opt-in via `SolveCallbacks::branch`.

## The Nogoods constraint

`gcs/constraints/nogoods.hh`. A nogood is a forbidden conjunction of decision
literals; the constraint forbids them all at once.

```cpp
using Nogood = std::vector<IntegerVariableCondition>;   // = / != / >= / <= literals
```

`Nogoods` **generalises `NegativeTable`**: a forbidden tuple is a nogood of
`var == value` literals, and `NegativeTable`'s entailment-based two-watched-
literals carries straight over to arbitrary `=`/`≠`/`≥`/`≤` literals.

- **Entailment-based 2WL.** A watched literal is "broken" when it is *entailed*,
  read via `State::test_literal` (`DefinitelyTrue`/`DefinitelyFalse`/`Undecided`):
  `x >= 10` is entailed once `lower(x) >= 10`, so `x >= 12` triggers it too. This
  is the bug class the per-node UP reference test (below) targets. When a clause
  is unit, the negation of the surviving literal is inferred (`JustifyUsingRUP`
  with `generic_reason` over the nogood's variables).
- **Consistency = unit propagation per clause**, *not* GAC on the conjunction
  (full CNF propagation is co-NP-hard).
- **NogoodStore** (`shared_ptr<vector<Nogood>>` + per-nogood variable lists) is
  the growable, **engine-owned** backing store. When restarts are enabled,
  `solve_with` installs one empty `Nogoods` over the store before the model is
  finalised, holds the store, and threads a `NogoodStore*` into the recursion.
  It is not user-posted — for parallel each thread owns its own store, so exposing
  it would be messy. The propagator's watches are **non-backtrackable** and grown
  lazily (`init_watches_for` catches up any nogoods learned since the last fire).

### Triggers caveat (performance)

v1 subscribes the `Nogoods` propagator to **every** variable
(`problem.all_normal_variables()`), and its propagator scans the whole store on
each wake. This is correct but degenerate: measured on `tsp`/`qap` it is a pure
*rec/s* slowdown (recursions unchanged) that scales with store size — the
propagator is woken on every domain change and does O(store) non-effectful work
per wake. The fix (per-variable/per-literal watch lists so a change visits only
the nogoods watching it, and the trigger set shrinks to watched variables) is a
**separate, larger planned stage**: "better propagation-queue triggers". It is
**semantics-preserving** — it changes only wall-clock time, not which nodes are
visited, which nogoods are learned, or the proof — so it is sequenced last.

## Reduced-nld extraction

`solve.cc`, the learning block. When a restart cutoff unwinds through a frame,
each sibling whose subtree was *fully refuted before the cutoff* (`refuted_siblings`)
yields a nogood.

The recorded clause is the **reduced** nld-nogood (Lecoutre et al., IJCAI 2007):
the *positive* decisions on the path to the frame, plus the refuted decision —
**not** the full path. A binary branch's second sibling is the negation of its
first (`x==v` then `x!=v`, or `x<=v` then `x>v`), so descending into it is a
*refutation flip*, a negative decision; it is **dropped**, because the first
sibling's own recorded nogood re-derives it on the next pass.

This is threaded as a `reduced_prefix` (a `vector<IntegerVariableCondition>` of
the kept positive decisions) passed down the recursion. A descended sibling is a
droppable negative iff it is the second sibling and equals the negation of the
first:

```cpp
bool negative_flip = sibling_index == 1 && first_sibling && guess == ! *first_sibling;
```

This is robust across literal kinds: `!(x<=v)` is `{GreaterEqual, v+1}` = `x>v`,
`!(x==v)` is `x!=v`, and `VariableConditionFrom` has a defaulted `<=>` so `==`
works. d-way values (`x==v1`, `x==v2`, …) are free positive decisions and are all
kept, so pure d-way search (e.g. `langford` with `smallest_first`) is unchanged.
The prefix is only maintained when nogood learning is active, so ordinary search
pays nothing.

A misdetection is **safe**: a false positive (dropping a literal that is *not*
re-derivable) produces an unsound clause that fails RUP in the proof — loud, not
silent — and a false negative just leaves the nogood less reduced.

Measured shrink (qap `--size=8`, `smallest_in`, same nogood counts, both
VeriPB-verified): `luby×50` 130→48 literals (2.7×), `luby×200` 86→20 (4.3×).
Deeper nogoods drop more negatives, so the shrink grows with restart depth.

**Interval and reject branches.** An *accept* interval decision (`x in a..b`) is
two bound literals `x>=a ∧ x<=b` — no disjunction, handled. A *reject* interval
(`x ∉ [a,b]`) is disjunctive; representing it as one clause literal would need an
interval-hole reified Boolean, the forbidden disjunctive reification that breaks
backtrack-from-guess RUP. No reject-interval value generator exists today, so it
is moot for v1 (the not-a-plain-condition guard in the extraction is a
conservative fallback). The range/interval-literal design from PR #303 makes
these literals first-class and dissolves that RUP wall, so reject branches become
representable once it lands.

## The proof lifecycle

This is the subtle part. Definitions and learned nogoods live at `ProofLevel::Top`
(index 0, never forgotten); each search frame `depth` works at proof level
`depth + 1`; a normal backtrack asserts `¬(guesses)` (RUP from the refutation) and
forgets level `depth + 1`.

Three things are non-obvious:

1. **Restart unwind keeps the root's level 1.** On a `RestartCutoffHit` unwind we
   assert *no* backtrack lemma — the subtree was abandoned unrefuted, so
   `¬(guesses)` is not RUP — we just forget what the pass derived. **But** the
   root node's guess-*independent* propagation lives at level 1, and the next pass
   starts from the same root fixpoint so `propagate()` re-emits nothing there.
   Forgetting it would make the next pass's backtracks non-RUP. So at the root
   (`depth == 0`) we keep level 1: `if (depth > 0) logger->forget_proof_level(depth + 1)`.
   Deeper guess-dependent levels are re-derived each pass.

2. **Reduced nogoods are RUP because of the unwind order.** A reduced clause drops
   the negative path decisions, so it is *not* RUP from Top facts alone — those
   dropped negatives must be re-derived. They are: the learning happens during the
   **deep-first unwind**, when the ancestor frames' backtrack lemmas (which force
   exactly those negatives — e.g. the `x==v` refutation that forces `x!=v`) are
   still live below Top. RUP uses all active levels, not just Top, so it
   re-derives the negatives and reaches the conflict. The clause is then emitted at
   Top (`ProofLogger::emit_learned_nogood`, a `% learned nogood` comment plus a
   `rup` line) so it survives this restart's forget and the whole loop.

3. **Enumeration needs no solution-blocking nogoods.** Complete enumeration logs
   each solution with `solx`, which *excludes* it (adds its negation to the proof
   database). So a fully-explored subtree — solutions and all — has every leaf a
   conflict (a genuine dead end or an excluded solution), which makes the plain nld
   nogood for it RUP, exactly like a real refutation's. That nogood stops a later
   pass re-entering and re-counting, so each solution is found, counted, and
   `solx`'d exactly once. This is why `solve_with_state` can keep searching past a
   solution while restarting, with no special blocking clause.

## Testing

The two-sided net (companion to the BinPacking per-bin pattern, *not*
`solve_for_tests_checking_gac`, which would wrongly flag a correct per-clause-UP
propagator):

- **Propagation core** (`gcs/constraints/nogoods_test.cc`): a standalone program
  feeding a fixed synthetic nogood set, with a per-node UP **reference** in the
  trace callback that asserts the 2WL invariant — an unsatisfied clause has `>= 2`
  non-entailed literals — using its own bound-based entailment so it catches a
  mis-computed one. Verified non-vacuous by weakening the propagator. Plus
  `--prove` for over-pruning/soundness.
- **End to end** (`gcs/solve_test.cc`): restart cases for unsat, optimisation,
  complete enumeration, root propagation (the level-1 fix), and — exercising the
  reduced extraction the d-way default leaves untouched — binary (`smallest_in`)
  and interval (`split_smallest_first`) branching. Each runs VeriPB; an unsound
  reduction fails RUP.
- `gcs/restarts_test.cc` checks the Luby sequence.

Proofs are the soundness guarantee throughout: any unsound learned clause, broken
entailment, or over-broadened nogood fails RUP rather than silently corrupting the
result.
